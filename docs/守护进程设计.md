# PrismPanel 守护进程功能与实现设计

本文描述 PrismPanel 守护进程的完整功能、数据格式、通信方式和关键实现规则。守护进程部署在 Minecraft 节点上，负责进程、控制台、文件和镜像服；面板负责用户权限、操作审计和请求转发。

设计参考仓库中的 MCSManager daemon 4.16.2：采用其节点监听、面板主动连接、主密钥认证、临时任务凭证、实例本地 JSON、控制台缓冲和整文件编辑等成熟思路。PrismPanel 根据本项目需求改用原生 WebSocket，增加普通实例与镜像服统一模型、批量文件树、文件版本冲突、部署事务、实时部署日志以及取消/强制结束语义。

---

## 一、总体架构

### 1.1 组件关系

```text
前端
├── 连接面板：登录、权限校验、操作审计、控制命令
└── 直连守护进程：控制台日志、部署日志、文件内容传输

面板
├── 使用守护进程主密钥建立管理 WebSocket
├── 校验用户对节点、服务器和文件的权限
├── 转发所有会改变进程状态的操作
├── 转发控制台输入、文件移动、重命名和删除
└── 为允许直连的功能签发临时凭证

守护进程
├── 提供 HTTP(S) API 和原生 WebSocket
├── 管理普通单实例和镜像服务器组
├── 管理 Java 进程、控制台、文件和部署任务
├── 在本地 JSON 文件中保存服务器配置
└── 向所有已认证面板广播状态和操作结果
```

守护进程允许多个面板同时连接。当前不区分面板身份，所有面板使用同一个守护进程主密钥；各面板自行维护用户、权限和审计数据。

### 1.2 请求链路

| 功能                            | 请求链路 |
|-------------------------------|---|
| 启动、停止、重启、强制关闭                 | 前端 -> 面板鉴权/审计 -> 守护进程 |
| 控制台命令                         | 每条命令均经过面板鉴权并记录，再转发给守护进程 |
| 控制台日志                         | 面板授权后，前端通过只读 WebSocket 直连守护进程 |
| 部署操作、取消、强制结束                  | 前端 -> 面板鉴权/审计 -> 守护进程 |
| 部署日志                          | 面板授权后，前端通过只读 WebSocket 直连守护进程 |
| 文件树、文件读取、上传、下载、编辑、文件移动、重命名、删除 | 面板签发临时凭证，前端通过 HTTP(S) 直连守护进程 |

移动和重命名复用同一个移动操作。重命名等价于源路径和目标路径位于同一目录，但目标文件名不同。

## 二、网络与协议

### 2.1 监听和访问地址

守护进程只配置一个监听地址、一个端口和一个可选的公网访问地址：

```yaml
server:
  listen: 0.0.0.0
  port: 24444
  public_url: https://node.example.com:24444
```

- `listen` 默认是 `0.0.0.0`，表示监听所有 IPv4 网卡。
- `0.0.0.0` 只能用于监听，不能作为浏览器或面板的访问地址。
- 面板添加节点时单独填写连接守护进程的 URL，例如内网地址 `http://192.168.1.20:24444`。
- 面板可以使用 IP 或域名连接守护进程。
- `public_url` 只用于前端直连日志和文件接口，不用于监听，也不用于面板管理连接。
- 守护进程不自动探测公网 IP，也不判断访问者是否位于内网。
- `public_url` 允许为空。为空时禁用前端直连，由面板代理对应的 WebSocket 和 HTTP 请求。

面板连接地址和公网地址可以不同：

```text
面板管理连接：http://192.168.1.20:24444
前端公网直连：https://node.example.com:24444
```

### 2.2 HTTP 与 HTTPS

守护进程原生支持 HTTP 和 HTTPS，通过一个布尔配置选择，二者不能在同一监听端口同时启动：

```yaml
ssl:
  enabled: true
  cert_file: /etc/prismpanel/fullchain.pem
  key_file: /etc/prismpanel/private.key
```

- `ssl.enabled=false`：监听端口提供 HTTP 和 WS。
- `ssl.enabled=true`：监听端口提供 HTTPS 和 WSS。
- HTTPS 模式启动时必须成功读取证书和私钥，否则守护进程启动失败，不自动降级到 HTTP。
- `public_url` 的协议必须与前端实际访问方式一致。
- HTTPS 面板页面不能直接访问 HTTP/WS 守护进程；浏览器通常会阻止这种混合内容。

### 2.3 协议组成

守护进程不使用 Socket.IO，使用原生 WebSocket 和普通 HTTP API：

```text
/api/v1/ws/control        面板管理连接
/api/v1/ws/console        前端只读控制台日志
/api/v1/ws/deployments    前端只读部署日志
/api/v1/files/*           前端文件接口
```

WebSocket 控制消息使用 JSON。文件内容直接使用 HTTP 请求体或响应体，不编码进 WebSocket JSON。

### 2.4 面板管理 WebSocket

连接建立后，面板必须在限定时间内发送认证消息：

```json
{
  "type": "auth",
  "secret": "daemon-main-secret"
}
```

认证失败或超时未认证时立即断开。认证成功后，面板可以发送带请求 ID 的命令：

```json
{
  "type": "instance.start",
  "request_id": "req-01",
  "data": {
    "server_id": "bedwars",
    "slot": 1
  }
}
```

响应统一关联原请求：

```json
{
  "type": "response",
  "request_id": "req-01",
  "success": true,
  "data": {}
}
```

实例状态、部署结果等事件广播给所有已认证的面板连接。连接使用 WebSocket Ping/Pong 检测断线，并限制单条消息大小和待发送队列长度。

## 三、认证和临时凭证

### 3.1 守护进程主密钥

- 守护进程首次启动时生成一个高强度随机主密钥。
- 主密钥保存在本地受限文件中，不由用户手工创建。
- 主密钥仅用于面板管理 WebSocket，绝不返回给前端。
- 守护进程提供查看和重置密钥的本地管理命令。
- 重置密钥后，现有面板管理连接失效，面板需要更新节点配置。

### 3.2 临时凭证流程

允许前端直连的功能使用短期、权限受限的临时凭证：

```text
前端向面板申请操作
-> 面板校验用户权限并记录待执行操作
-> 面板通过管理 WebSocket 向守护进程注册临时凭证
-> 面板向前端返回 public_url、凭证和过期时间
-> 前端携带凭证直连守护进程
-> 守护进程只执行凭证允许的操作
-> 守护进程把最终结果回报面板
```

凭证至少绑定：

- 随机 `ticket_id`；
- 实例 ID 或部署任务 ID；
- 权限范围；
- 允许访问的文件路径或目录；
- 文件大小上限；
- 过期时间；
- 可用次数或会话属性。

典型权限范围：

```text
console.read
deployment.read
file.list
file.read
file.upload
file.download
file.edit
```

日志 WebSocket 连接建立后，在首条消息中提交凭证，避免凭证进入 URL 和访问日志。文件 API 使用 `Authorization: Bearer <ticket>`。

### 3.3 编辑凭证续期

文件编辑凭证绑定确切实例和文件路径，可在有效期内多次读取、保存。前端必须保存 `expires_at`：

1. 保存前发现凭证即将到期时，向面板重新申请。
2. 守护进程返回 `TICKET_EXPIRED` 时，前端保留编辑器内容并重新申请。
3. 面板重新校验用户当前权限后签发新凭证。
4. 获得新凭证后，前端只自动重试一次明确未执行的保存请求。
5. 网络超时导致结果不确定时，不盲目重复覆盖，先查询文件版本。

关闭编辑器时可以主动撤销凭证；撤销请求失败时，凭证仍会在到期后自动失效。

## 四、本地配置和存储

守护进程不使用数据库，所有持久数据保存在本地文件中。

### 4.1 目录结构

```text
data/
├── daemon.yaml                         # 守护进程全局配置
├── secret.json                         # 自动生成的主密钥
├── servers/                            # 服务器配置
│   ├── lobby.json
│   ├── proxy.json
│   └── bedwars.json
└── runtime/
    └── deployments/
        ├── <task-id>.json              # 部署任务状态
        └── <task-id>.jsonl             # 部署任务日志
```

守护进程启动时扫描 `servers/*.json` 加载服务器，不维护包含所有完整配置的单一数组文件。一个配置损坏时只隔离对应服务器，不影响其他服务器。

配置保存使用同目录临时文件和原子重命名，避免进程退出或断电留下半个 JSON 文件。每个 JSON 包含 `schema_version`，未知版本拒绝加载并返回明确错误。

实时进程状态、CPU、内存、控制台订阅和运行 PID 保存在内存中，不写入服务器配置。

### 4.2 全局配置

```yaml
server:
  listen: 0.0.0.0
  port: 24444
  public_url: null

ssl:
  enabled: false
  cert_file: null
  key_file: null

storage:
  data_dir: /var/lib/prismpanel

files:
  max_edit_file_size: 5242880

java_runtimes:
  java17: /opt/java/17/bin/java
  java21: /opt/java/21/bin/java
  java25: /home/java/25/bin/java
```

Java 环境由守护进程全局维护。实例只引用 Java ID，例如 `java25`。路径必须指向 Java 可执行文件，建议使用绝对路径；启动实例前检查文件存在、是普通文件且具有执行权限。

### 4.3 服务器 ID

- `server_id` 在整个守护进程内全局唯一，创建后不可修改。
- 配置文件名是 `servers/<server_id>.json`。
- 仅允许小写字母、数字、`-` 和 `_`。
- 镜像派生实例 ID 为 `<server_id>_<slot>`。
- 新建服务器时同时检查其 ID 是否与已有镜像派生实例 ID 冲突。
- 展示名称 `name` 可以修改，不参与路径或实例定位。

## 五、服务器模型

守护进程支持普通单实例和镜像服务器组，两者复用相同的进程、控制台、文件和监控实现。

### 5.1 普通单实例

```json
{
  "schema_version": 1,
  "type": "standalone",
  "server_id": "lobby",
  "name": "主城服",
  "workspace": "/srv/minecraft/lobby",
  "port": 25565,
  "runtime": {
    "java_runtime": "java21",
    "server_jar": "server.jar",
    "min_memory_mib": 1024,
    "max_memory_mib": 4096,
    "jvm_args": [],
    "server_args": ["nogui"]
  },
  "process": {
    "stop_command": "stop",
    "stop_timeout_seconds": 60,
    "auto_start": false,
    "auto_restart": true
  },
  "console": {
    "encoding": "utf-8"
  }
}
```

普通实例的实例 ID 等于 `server_id`。端口由配置管理，并写入 `server.properties` 的 `server-port`。

### 5.2 镜像服务器组

```json
{
  "schema_version": 1,
  "type": "mirror",
  "server_id": "bedwars",
  "name": "起床战争",
  "root_path": "/srv/minecraft/bedwars",
  "image_directory": "image",
  "instance_count": 3,
  "ports": [25571, 25572, 25573],
  "exclude": [
    {
      "path": "world",
      "type": "directory"
    },
    {
      "path": "server.properties",
      "type": "file"
    }
  ],
  "runtime": {
    "java_runtime": "java21",
    "server_jar": "server.jar",
    "min_memory_mib": 1024,
    "max_memory_mib": 4096,
    "jvm_args": [],
    "server_args": ["nogui"]
  },
  "process": {
    "stop_command": "stop",
    "stop_timeout_seconds": 60,
    "auto_start": false,
    "auto_restart": true
  },
  "console": {
    "encoding": "utf-8"
  }
}
```

目录派生规则：

```text
/srv/minecraft/bedwars/image              镜像目录
/srv/minecraft/bedwars/bedwars_1          第 1 个实例
/srv/minecraft/bedwars/bedwars_2          第 2 个实例
/srv/minecraft/bedwars/bedwars_3          第 3 个实例
```

端口映射规则：

```text
bedwars_1 -> ports[0]
bedwars_2 -> ports[1]
bedwars_3 -> ports[2]
```

约束如下：

- `ports` 数量不得小于 `instance_count`。
- 守护进程内所有受管实例端口必须全局唯一。
- 端口范围是 1-65535，不能与守护进程监听端口冲突。
- `image_directory` 必须位于 `root_path` 内，不能与派生实例目录冲突。
- `exclude.path` 必须是相对路径，禁止绝对路径和 `..`。
- `exclude.type` 必须明确为 `file` 或 `directory`；排除目录时保留其整个子树。
- 同一份排除列表同时用于首次部署和后续同步。

`instance_count` 从 3 减少到 2 时，`bedwars_3` 不停止、不删除、不移动；后续部署、状态管理和代理服同步只处理 `bedwars_1`、`bedwars_2`。数量恢复为 3 时，现有 `bedwars_3` 重新纳入管理。

### 5.3 代理服地址

普通实例和镜像实例都保存端口。面板同步到 BungeeCord/Velocity 时，使用面板连接守护进程 URL 的主机部分加实例有效端口：

```text
面板连接地址：http://192.168.1.20:24444
lobby：       192.168.1.20:25565
bedwars_1：   192.168.1.20:25571
```

守护进程不额外保存游戏服务地址。协议、管理端口、URL 路径和查询参数不参与代理服地址生成。

## 六、实例进程管理

### 6.1 实例状态

```text
stopped     已停止
starting    正在启动
running     进程正在运行
stopping    正在正常关闭或等待超时
deploying   正在执行镜像部署
failed      最近一次启动、停止或部署失败
```

`running` 只表示 Java 进程存在，不代表 Minecraft 已完成加载。TPS、在线玩家和游戏内就绪状态由后续 Minecraft 插件或专用查询功能提供。

同一实例的启动、停止、重启、强制关闭、配置应用和部署操作必须串行执行。实例处于过渡状态时，冲突操作返回 `INSTANCE_BUSY`，不能启动第二个进程。

### 6.2 启动

启动前依次执行：

1. 校验服务器配置和实例槽位。
2. 解析全局 `java_runtimes` 中的 Java ID。
3. 校验工作目录和 `server_jar`，拒绝越出工作目录的路径和符号链接。
4. 检查配置端口是否与其他受管实例和守护进程端口冲突。
5. 将配置端口写入 `server.properties`。
6. 使用结构化参数构造 Java 命令，不通过 Shell 拼接执行。
7. 创建 stdin、stdout、stderr 管道和独立进程组。
8. 进程创建成功后进入 `running`，开始控制台转发和指标采集。

启动命令由以下字段构造：

```text
<java_runtime> <memory arguments> <jvm_args> -jar <server_jar> <server_args>
```

实例配置不保存可任意执行的 Shell 命令。运行 PID 只用于当前守护进程会话，不能在守护进程重启后仅凭旧 PID 接管或杀死进程。

### 6.3 正常关闭和强制关闭

```text
状态进入 stopping
-> 向 stdin 写入 stop_command 和换行符
-> 等待 stop_timeout_seconds
-> 超时后强制结束整个实例进程组
-> 等待进程退出并释放管道
-> 状态进入 stopped
```

- `stop_command` 默认是 `stop`。
- 关闭超时后直接 kill，不恢复为 `running`。
- 强制关闭跳过 stdin 和等待阶段，直接结束整个进程组。
- 停止和强制关闭作用于 `stopped` 时按幂等成功处理。
- 关闭过程中抑制自动重启。

### 6.4 重启和自动重启

手动重启等价于正常关闭完成后重新启动。关闭超时触发 kill 后，只要确认旧进程已退出，手动重启仍继续启动。

`auto_restart=true` 时，非用户操作导致的异常退出可以自动重新启动。自动重启需要有次数上限和退避，防止错误配置造成无限快速重启；用户停止、部署停止和守护进程退出不触发自动重启。

### 6.5 守护进程退出

守护进程收到退出信号后停止接受新的启动和部署任务，依次对运行实例执行正常关闭；达到全局退出期限后强制结束剩余实例进程组。退出前关闭 WebSocket、刷新任务状态并释放本地锁。

## 七、配置保存和应用

面板修改服务器配置时提供两个操作：`完成` 和 `部署`。

### 7.1 点击完成

1. 完整校验新配置，包括服务器 ID、路径、Java ID、实例数量和端口冲突。
2. 原子保存对应的服务器 JSON。
3. 立即修改当前受管实例的 `server.properties`，写入配置端口。
4. 不停止、不重启实例，也不复制镜像文件。

如果实例正在运行，新的端口只写入文件，不会改变当前 Java 进程正在监听的端口，直到下次重启或部署才生效。因此状态查询应区分：

```json
{
  "configured_port": 25572,
  "file_port": 25572,
  "runtime_port": 25571,
  "pending_restart": true
}
```

- `configured_port`：服务器 JSON 中的目标端口。
- `file_port`：当前 `server.properties` 中的端口。
- `runtime_port`：当前运行进程启动时使用的端口；停止时为 `null`。
- `pending_restart`：运行端口与配置端口不一致。

代理服同步在实例运行期间使用 `runtime_port`，实例下次启动成功后再切换到 `configured_port`。如果实例目录或 `server.properties` 不存在，JSON 仍然保存；文件配置标记为待应用，在下次启动或部署时创建并写入。

普通实例只修改自己的 `server.properties`。镜像服务器组按 `1..instance_count` 依次修改对应实例，不处理数量范围外的旧目录。

### 7.2 点击部署

- 镜像服务器组：保存 JSON 后立即创建镜像部署任务，按部署流程同步所有当前实例。
- 普通单实例：保存 JSON、记录原运行状态、关闭实例、写入 `server.properties`，然后按原运行状态重新启动；不执行镜像复制。

### 7.3 修改 server.properties

守护进程只修改 `server-port`：

- 存在该字段时替换值。
- 不存在时在文件末尾追加。
- 文件不存在时创建文件并写入该字段。
- 保留其他字段、注释、顺序和原换行风格。
- 写入使用临时文件和原子替换。
- 启动和镜像部署时再次校验并修正端口。

## 八、控制台

守护进程读取实例 stdout 和 stderr，写入每个实例独立的有界环形缓冲，并向已认证的前端订阅者推送。控制台日志不经过面板，因此不会增加面板的日志转发负载。

控制台帧包含实例 ID、进程会话 ID、递增序号、stdout/stderr 类型、时间和原始内容。每次启动生成新会话 ID；新订阅者可获取缓冲中的最近输出。缓冲和发送队列均有全局上限，慢客户端不能阻塞 Java stdout；队列溢出时推送缺口事件。

每条控制台命令必须经过面板：面板检查用户的控制台写入权限，记录用户、实例、命令和时间，再通过管理 WebSocket 转发。守护进程校验实例状态和命令长度后写入 Java stdin。

控制台只读临时凭证不能发送命令。单条命令禁止 NUL 字节并限制最大长度，守护进程负责追加实例配置的换行符。

## 九、文件系统

### 9.1 路径边界

所有文件操作绑定某个实例，只允许访问实例工作目录内的相对路径。守护进程拒绝绝对路径和 `..`，解析真实父路径以防止符号链接逃逸，并保证源路径、目标路径都位于同一实例工作目录。文件接口还需限制请求体大小、文件数量、并发任务和超时。

### 9.2 文件树批量目录接口

前端使用可展开、可折叠的文件树。列表采用懒加载：折叠仅改变前端状态；展开时请求该目录；刷新时一次提交当前所有已展开目录。

```http
POST /api/v1/files/list
Authorization: Bearer <temporary-ticket>
```

```json
{
  "instance_id": "bedwars_1",
  "directories": [
    { "path": ".", "cursor": "", "limit": 200 },
    { "path": "plugins", "cursor": "", "limit": 200 },
    { "path": "world/datapacks", "cursor": "", "limit": 200 }
  ],
  "sort": {
    "field": "name",
    "order": "asc",
    "directories_first": true
  },
  "include_hidden": true
}
```

响应按目录分别返回：

```json
{
  "results": [
    {
      "path": "plugins",
      "entries": [
        {
          "name": "PrismPlugin.jar",
          "path": "plugins/PrismPlugin.jar",
          "type": "file",
          "size": 182340,
          "modified_at": "2026-07-13T11:00:00Z",
          "mode": 420
        }
      ],
      "next_cursor": "",
      "truncated": false,
      "error": null
    }
  ]
}
```

一个目录读取失败时在该结果的 `error` 中返回错误，不影响批次中的其他目录。大目录使用不透明 `cursor` 分页；服务端限制一次请求的目录数量、单目录条目数和响应总条目数。目录默认优先于文件并按名称升序。

无需递归预查目录是否包含子项。前端先允许展开，展开后得到空数组时再显示为空目录。

### 9.3 上传和下载

上传和下载都在面板鉴权后由前端直连守护进程：

- 下载凭证绑定确切实例和文件路径，默认单次使用。
- 上传凭证绑定确切实例和目标目录，并包含单文件/总大小、文件数量和覆盖策略限制。
- 文件内容不经过面板，但守护进程将最终成功或失败结果通过管理连接报告面板。
- 上传先写临时文件，接收完成并校验大小后原子移动到目标位置。
- 下载前再次验证凭证路径与真实文件路径。

### 9.4 文件编辑

文件编辑采用与 MCSManager 相同的整文件模式，不按行切片：

```text
面板校验权限并签发编辑凭证
-> 前端直连守护进程打开文件
-> 守护进程检查全局大小上限
-> 未超限则整文件读取并返回
-> 前端编辑
-> 保存时整文件提交并原子覆盖
```

全局配置 `files.max_edit_file_size` 默认是 5 MiB。读取和保存都不能超过该限制，面板和临时凭证不能提高上限。

读取响应包含 `path`、`content`、`encoding`、`size`、`modified_at` 和 `version`。保存时提交 `path`、完整 `content` 和 `expected_version`。如果文件在编辑期间被其他程序修改，返回 `FILE_CHANGED` 并拒绝覆盖；前端保留未保存内容，提示重新加载或人工处理冲突。

编辑凭证到期时按第三章的续期流程重新申请，不清空编辑器内容。

### 9.5 移动、重命名和删除

这些操作请求体很小，全部经过面板权限校验和审计，再通过管理 WebSocket 转发，不使用前端直连凭证。

移动：

```json
{
  "instance_id": "bedwars_1",
  "source": "plugins/a.jar",
  "destination": "plugins-disabled/a.jar",
  "overwrite": false
}
```

重命名复用移动，只需将 `destination` 改为同目录下的新文件名。删除支持文件和目录列表；目录递归删除必须由请求显式声明，守护进程不能根据目标类型静默扩大删除范围。

## 十、镜像部署

### 10.1 部署目标

部署支持全部实例、单个实例和多个指定实例：

```json
{ "server_id": "bedwars", "targets": "all" }
```

```json
{ "server_id": "bedwars", "targets": [1, 2] }
```

`all` 解析为 `1..instance_count`。槽位必须位于当前范围内。批量部署严格按槽位顺序执行，不并行复制和重启实例。

### 10.2 部署互斥

同一个镜像服务器组同一时间只能存在一个部署任务。即使当前任务只部署 `bedwars_1`，也不能同时部署 `bedwars_2`。再次提交时返回 `DEPLOYMENT_ALREADY_RUNNING` 和现有 `task_id`。

不同 `server_id` 可以并行部署，但仍受守护进程全局部署并发和磁盘 I/O 限制。

### 10.3 批量预检查

开始关闭任何实例前，统一检查：

- 镜像目录和 `server.jar` 存在且可读。
- 实例数量、目标槽位和端口列表合法。
- 排除项类型和路径合法。
- 目标端口无配置冲突。
- 根目录、镜像目录、实例目录和临时目录均位于允许路径内。
- 磁盘空间满足部署要求。
- 没有未恢复的部署临时目录或备份目录。

预检查失败时一个实例都不关闭。

### 10.4 单实例部署事务

假设目标是 `bedwars_1`，在镜像根目录中创建同级临时目录：

```text
/srv/minecraft/bedwars/
├── image/
├── bedwars_1/
├── .deploy-bedwars_1-<task-id>/
└── .backup-bedwars_1-<task-id>/
```

完整流程：

1. 获取实例操作锁并记录部署前是否运行。
2. 如果正在运行，发送 `stop_command` 并等待关闭超时。
3. 超时后强制结束整个进程组；确认完全退出后才继续。
4. 创建空的 `.deploy-*` 临时目录。
5. 从 `image` 复制非排除内容到临时目录。
6. 从原实例目录复制排除的文件和目录到临时目录。
7. 修改临时目录 `server.properties` 中的 `server-port`。
8. 校验临时目录中的必要文件和配置。
9. 将原实例目录重命名为 `.backup-*`。
10. 将 `.deploy-*` 原子重命名为正式实例目录。
11. 删除备份目录。
12. 如果部署前正在运行，则启动新版本；原来停止则保持停止。

镜像、实例、临时目录和备份目录应处于同一文件系统，确保目录重命名具备原子语义。部署期间禁止对目标实例执行文件写入、移动、删除和其他生命周期操作。

### 10.5 排除规则

同步后的目标语义是：

```text
排除项：保留实例原有内容，既不删除，也不从 image 覆盖
其他项：删除实例旧内容，以 image 中的内容为准
```

实现使用“镜像复制到临时目录 + 从旧实例恢复排除项”，而不是直接在正式目录中先删除再复制。这样复制失败时旧实例保持完整。

- 排除 `world` 目录时保留整个 `world` 子树。
- 排除项在旧实例中不存在时保持不存在，即使镜像中存在同名项也不复制。
- 排除项声明为文件但实际是目录，或声明为目录但实际是文件时，部署失败。
- 即使 `server.properties` 被排除，恢复后仍强制修改 `server-port`，其余字段保持实例原内容。

### 10.6 失败处理

- 复制、排除恢复、端口修改或校验失败：删除临时目录，保留旧实例目录。
- 目录已经交换后发生失败：使用备份目录回滚。
- 部署文件失败时，若部署前实例正在运行，重新启动旧版本。
- 文件替换成功但新版本启动失败：保留新文件，返回“部署成功、启动失败”，不自动回滚文件。
- 单个实例失败后记录结果并继续部署下一个实例。
- 回滚失败时标记 `FAILED_NEEDS_ATTENTION`，不自动启动，也不允许创建新部署任务，直到人工处理残留目录。

单实例结果分别报告停止、文件部署、端口写入和重新启动结果，避免用一个布尔值隐藏具体失败阶段。

### 10.7 数量减少

`instance_count` 减少时，多余实例目录和进程不做任何处理。部署只处理当前 `1..instance_count`；代理服同步也不再包含范围外实例。恢复数量后，原目录重新纳入管理。

## 十一、部署任务日志

### 11.1 创建和订阅

部署是异步任务。面板发起部署后，守护进程立即返回 `task_id`。前端经面板授权后，使用只读临时凭证连接：

```text
ws(s)://<public-url>/api/v1/ws/deployments
```

WebSocket 首条消息：

```json
{
  "type": "auth",
  "ticket": "temporary-ticket",
  "task_id": "deploy-xxx",
  "after_sequence": 0
}
```

凭证只允许读取指定任务日志，不能部署、取消或强制结束任务。`public_url` 为空时，由面板代理相同的 WebSocket 消息，前端协议不变。

### 11.2 日志消息

每条日志具有递增 `sequence`：

```json
{
  "type": "log",
  "task_id": "deploy-xxx",
  "sequence": 12,
  "timestamp": "2026-07-13T14:20:31Z",
  "level": "info",
  "server_id": "bedwars",
  "instance_id": "bedwars_1",
  "stage": "copying_image",
  "message": "正在复制镜像文件"
}
```

稳定阶段枚举：

```text
queued
prechecking
stopping
force_killing
preparing
copying_image
restoring_excluded
writing_port
validating
swapping
starting
rolling_back
completed
failed
```

复制过程推送文件数和字节数的汇总进度，不为每个成功文件发送日志；具体路径只在失败时记录。

### 11.3 断线恢复

守护进程为每个任务保存有界日志和最终结果。前端重连时携带最后收到的 `after_sequence`，守护进程先补发后续历史日志，再继续实时推送。任务状态和日志按全局保留期限清理；长期用户审计由面板保存。

### 11.4 任务状态

```text
queued
running
cancel_requested
force_stop_requested
rolling_back
cancelled
force_stopped
completed
completed_with_errors
failed
failed_needs_attention
```

守护进程同时提供任务状态查询，以便前端首次打开或 WebSocket 断开时获得当前槽位、阶段、总体进度和各实例结果。

## 十二、取消和强制结束部署

取消和强制结束必须经过面板权限校验和操作审计，部署日志凭证没有终止权限。

### 12.1 普通取消

普通取消在安全点生效：

1. 面板向守护进程发送 `cancel(task_id)`。
2. 任务进入 `cancel_requested`。
3. 当前复制批次或目录操作到达安全点后停止。
4. 删除临时目录或完成旧目录回滚。
5. 恢复当前实例部署前的运行状态。
6. 不再部署后续实例。
7. 标记 `cancelled` 并释放服务器组部署锁。

如果正在执行目录交换，取消必须等待交换或回滚完成，不能让正式实例目录停留在不存在的中间状态。

### 12.2 强制结束

强制结束表示尽快中断当前部署：

1. 立即停止读取和复制新数据，关闭可中断的文件流。
2. 删除未完成的临时目录。
3. 正式目录尚未替换时保留旧目录；已经交换时立即完成最小必要回滚。
4. 不恢复部署前运行状态，实例最终保持停止，由用户手动启动。
5. 不再部署后续实例。
6. 成功清理后标记 `force_stopped` 并释放锁。

强制结束不等于杀死守护进程，也不能中断不可拆分的原子重命名。若回滚失败，状态为 `failed_needs_attention`，禁止自动启动和再次部署，等待人工处理。

守护进程重启时扫描 `.deploy-*`、`.backup-*` 和未完成任务记录，完成安全恢复或标记需要人工处理后，才允许该服务器组重新部署。

## 十三、节点监控和状态上报

守护进程定期采集并通过管理 WebSocket 广播：

### 13.1 节点指标

- CPU 使用率和系统负载；
- 总内存、可用内存和使用率；
- 数据目录所在磁盘的总量、可用量和使用率；
- 操作系统、架构、守护进程版本和运行时间；
- 管理连接数、控制台订阅数和正在执行的部署任务。

### 13.2 实例指标

- 实例状态、PID 和运行时间；
- 主进程及其子进程树的 CPU 和内存；
- 配置端口、文件端口、运行端口和是否待重启；
- 最近启动、停止、退出码和错误原因；
- 当前部署任务和阶段。

面板连接断开不影响已运行实例。面板重新连接后，守护进程先发送节点与全部当前实例的完整状态快照，再发送增量事件。

TPS、在线人数、玩家列表和插件信息不由操作系统进程指标推断，后续通过 Minecraft 插件或查询协议获取。

## 十四、错误和操作结果

面板和前端逻辑必须根据稳定错误码处理，不解析错误文本。主要错误码：

| 错误码 | 含义 |
|---|---|
| `UNAUTHENTICATED` | 主密钥或临时凭证无效 |
| `TICKET_EXPIRED` | 临时凭证已过期，操作未执行 |
| `PERMISSION_DENIED` | 凭证不允许当前实例、路径或操作 |
| `SERVER_NOT_FOUND` | 服务器 ID 不存在 |
| `INSTANCE_NOT_FOUND` | 实例或槽位不存在 |
| `INSTANCE_BUSY` | 实例正在执行冲突操作 |
| `INVALID_STATE` | 当前状态不允许操作 |
| `PORT_CONFLICT` | 端口配置冲突 |
| `JAVA_RUNTIME_NOT_FOUND` | Java ID 不存在或不可执行 |
| `PATH_ESCAPE` | 路径越出实例工作目录 |
| `FILE_TOO_LARGE` | 文件超过编辑或传输限制 |
| `FILE_CHANGED` | 编辑期间文件已被其他程序修改 |
| `DEPLOYMENT_ALREADY_RUNNING` | 同一镜像组已有部署任务 |
| `DEPLOYMENT_CANCELLED` | 部署已在安全点取消 |
| `DEPLOYMENT_FORCE_STOPPED` | 部署已被强制结束 |
| `ROLLBACK_FAILED` | 部署目录无法自动恢复 |
| `PROCESS_START_FAILED` | Java 进程启动失败 |
| `PROCESS_STOP_FAILED` | 无法确认实例进程已退出 |
| `INTERNAL` | 未分类内部错误，附日志关联 ID |

文件直连操作和部署任务完成后，守护进程将最终结果通过所有已认证面板管理连接广播。面板根据 `request_id`、`ticket_id` 或 `task_id` 完成操作日志。

## 十五、关键并发与安全规则

- 同一实例的生命周期、部署和文件写操作串行执行。
- 同一镜像服务器组只能有一个部署任务；批量目标按顺序部署。
- 不使用 Shell 执行实例启动命令。
- 守护进程和 Java 实例默认以非 root 用户运行。
- 进程停止和强杀作用于完整进程组，避免残留子进程。
- 所有文件路径经过实例工作目录边界与符号链接检查。
- 主密钥不提供给前端，临时凭证遵循最小权限和自动过期。
- WebSocket、日志缓冲、部署日志、文件请求和异步任务全部有容量上限。
- JSON 配置和 `server.properties` 使用临时文件原子替换。
- 镜像部署使用同级临时目录和备份目录，失败时保留或恢复旧实例。
